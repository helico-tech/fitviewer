<feature>
  <name>FIT Data Pipeline: Parse, Normalize, Downsample</name>
  <files>

  <behavior>
    ## normalize.ts
    - semicirclesToDegrees(519828467) returns approximately 43.6 (within 0.001 tolerance)
    - semicirclesToDegrees(0) returns 0
    - semicirclesToDegrees(-519828467) returns approximately -43.6
    - The conversion factor is exactly 180 / 2**31

    ## fit-parser.ts
    - parseFitFile with a valid running FIT ArrayBuffer returns a NormalizedActivity with non-empty records, populated summary, and metadata.sport === 'running'
    - parseFitFile with random garbage bytes throws an Error containing "Not a valid FIT file"
    - parseFitFile with a valid cycling FIT file throws an Error containing "SPORT_MISMATCH:cycling" (or whatever sport is present)
    - parseFitFile returns records with positionLat/positionLong as decimal degrees (not raw semicircles -- values should be between -180 and 180)
    - parseFitFile returns records with cadence doubled (steps per minute, not half-cycles)
    - parseFitFile returns summary with totalDistance, totalDuration populated as numbers > 0
    - parseFitFile returns metadata with device as non-empty string, date as Date, sport as 'running'

    ## downsample.ts
    - downsampleTimeSeries with data.length <= targetPoints returns the original data unchanged
    - downsampleTimeSeries with data.length > targetPoints returns exactly targetPoints entries
    - downsampleTimeSeries preserves the first and last data points
    - downsampleTimeSeries with empty array returns empty array

    ## worker.ts
    - Worker receives { type: 'parse', buffer } and posts back { type: 'success', activity } on valid FIT data
    - Worker receives { type: 'parse', buffer } with invalid data and posts back { type: 'error', message, details }
  </behavior>
  <implementation>
    **Test fixture strategy:** To test fit-parser.ts properly, we need real FIT files. During the RED phase:
    1. Create `tests/fixtures/` directory
    2. Generate a minimal valid running FIT file by: install @garmin/fitsdk, use its Encoder to create a tiny FIT file with a few records (session with sport='running', 3-5 record messages with HR/cadence/GPS/speed/altitude, 1 lap message, fileId message). Save as `tests/fixtures/running.fit`.
    3. Generate a cycling FIT file the same way (session with sport='cycling'). Save as `tests/fixtures/cycling.fit`.
    4. Create `tests/fixtures/garbage.bin` -- 100 random bytes.
    5. Document the fixtures in `tests/fixtures/README.md`.

    If the @garmin/fitsdk Encoder is not available or too complex, alternative: download sample FIT files from the Garmin FIT SDK GitHub repo (they include sample files), or create a test that generates an ArrayBuffer using the known FIT binary header format.

    **Implementation order:**
    1. RED: Write all normalize.test.ts tests -> run (fail) -> commit
    2. GREEN: Implement normalize.ts -> run (pass) -> commit
    3. RED: Create test fixtures, write fit-parser.test.ts tests -> run (fail) -> commit
    4. GREEN: Implement fit-parser.ts following RESEARCH.md Pattern 4 exactly -> run (pass) -> commit
    5. RED: Write downsample.test.ts tests -> run (fail) -> commit
    6. GREEN: Implement downsample.ts using the `downsample` npm package LTTB -> run (pass) -> commit
    7. Implement worker.ts (the Web Worker entry point) following RESEARCH.md Pattern 3 -- imports parseFitFile, handles 'parse' messages, posts back success/error responses. Uses `satisfies WorkerResponse` for type safety on postMessage calls.

    **Key implementation details from RESEARCH.md:**
    - FIT parser: Use Decoder.isFIT() and Decoder.checkIntegrity() before decode. Use all decode options (applyScaleAndOffset, expandSubFields, expandComponents, convertTypesToStrings, convertDateTimesToDates, mergeHeartRates). Check session.sport !== 'running' and throw with SPORT_MISMATCH:{sport} prefix.
    - Semicircles: `degrees = semicircles * (180 / 2**31)`. This is NOT handled by applyScaleAndOffset.
    - Cadence: FIT stores half-cycles. Double record-level cadence. Session-level avgRunningCadence may already be doubled -- use it if available, otherwise double avgCadence.
    - Enhanced fields: Prefer enhancedSpeed over speed, enhancedAltitude over altitude.
    - Missing channels: Every field on ActivityRecord except timestamp is nullable. Null-check everything.
    - Transfer: Worker should note ArrayBuffer is transferred in (not copied).

    **Worker test note:** Worker.ts tests may need to be integration-level or tested indirectly through the worker message interface. If bun:test cannot instantiate Workers directly, test the underlying parseFitFile function thoroughly (which IS directly testable) and keep the worker.ts wrapper minimal and obvious enough to verify by inspection. Add a comment noting this.
